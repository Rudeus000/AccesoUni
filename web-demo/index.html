ERS v2.0 · Documento de requerimientos

Accesibilidad universal en portales universitarios

AccesoUni es una solución SaaS orientada a estudiantes con discapacidad visual, auditiva o cognitiva: activación por voz, lector semántico, ajustes visuales y métricas de cumplimiento para la institución. Compatible con extensión Manifest V3 e inyección por script en WordPress y Wix.

Marco normativo y técnico

  • WCAG 2.1 nivel AA
  • SUNEDU / MINEDU
  • Ley N.º 29733 — protección de datos personales (Perú)
  • IEEE 830 — especificación de requerimientos

Requerimientos funcionales (usuario)

Resumen del núcleo descrito en el ERS v2: valor para el estudiante y para la gestión institucional.

RU01 · Alta

Activación por voz

Comando despertador y Web Speech API para activar AccesoUni sin depender de la vista (hands-free).

RU02 · Alta

Lector semántico

Mapa auditivo de la página según jerarquía HTML5: regiones, encabezados y contenido principal.

RU03 · Alta

Subtitulado dinámico

Transcripción en tiempo real cuando el multimedia carece de subtítulos nativos (según capacidades del navegador y CORS).

RU04 · Media

Alertas visuales

Refuerzo visual perimetral para usuarios sordos ante eventos que antes solo eran sonoros.

RU05 · Alta

Capas visuales

Contraste, tamaño de texto hasta 200 %, interlineado, espaciado y modo de color vía CSS inyectado.

RU06 · Media

Perfil en la nube

Sincronización de preferencias con backend FastAPI y Supabase Auth cuando el usuario tiene sesión.

Características del sistema (CAR)

CAR01–CAR04

Experiencia inclusiva

Voz, lector semántico, subtítulos asistidos y ajuste visual completo con vista previa en sitio.

CAR05–CAR06

Datos y auditoría

Perfil sincronizado y escaneo heurístico por página con envío de métricas al API para reportes de cumplimiento.

CAR07–CAR08

Valor institucional

Panel SaaS y certificados PDF de cumplimiento para procesos ante SUNEDU (roadmap según despliegue).

CAR09–CAR10

Seguridad multi-tenant

RLS en Supabase, política de retención y aislamiento por institución descritos en el modelo de datos del ERS.

Arquitectura técnica (RS)

Cliente

Manifest V3

Service worker, content script y popup; sin código remoto arbitrario en la extensión publicada.

Embed

Widget widget.js

Misma lógica central inyectada en WordPress/Wix: npm run build en extension/ y copia automática a esta demo.

Servidor

FastAPI + Supabase

API REST, dominios permitidos, preferencias y logs de cumplimiento con PostgreSQL y RLS.
